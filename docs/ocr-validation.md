# OCR Validation

SAKEログのOCR、画像前処理、銘柄候補抽出、市場価格照合について、無料・ブラウザ内実装を前提に検証した記録です。

## 使用したOCR方式

- ブラウザ実装: `TextDetector` が使える場合は高速OCRとして先に試行
- フォールバック: `Tesseract.js` をOCR実行時に動的import
- 検証実行: ローカルに導入済みの Tesseract CLI で `jpn`, `eng`, `jpn_vert`, `jpn+eng` を比較
- 実装方針: OCR結果は自動確定せず、一致理由と信頼度を表示してユーザー確認を必須にする

## 使用した前処理

- EXIF Orientation補正
- 中央ラベル領域の切り出し
- 解像度調整と2倍拡大
- グレースケール化
- コントラスト補正
- 明るさ補正
- 二値化
- シャープ化
- OCR結果の正規化
- OCR誤認識文字の補正
- N-gram類似度
- レーベンシュタイン距離
- 蔵元名、容量、酒種との整合

## テスト画像条件

`scripts/Run-OcrValidation.ps1` で合成画像を生成し、`tests/fixtures/` に保存しました。

- 正面から撮影した表ラベル
- 裏ラベル
- ボトル全体相当
- 暗い場所
- 反射
- ピンぼけ相当
- 斜め撮影
- 縦書き文字
- 筆文字風の大きな日本語
- 小さい文字
- 日本語と英語混在
- 背景に他の文字がある
- 複数の瓶が写っている
- HEIC由来画像相当のJPEG
- JPEG
- PNG
- スクリーンショット相当
- SNS経由でEXIFが消えた画像相当

WebPはブラウザ実装では `image/webp` として受け付けますが、今回のTesseract CLI検証環境ではWebP直接入力を使わず、ブラウザ側の `createImageBitmap` 前提の対応として扱いました。

## 最終採用方式

アプリでは以下を採用しました。

1. TextDetectorを試す
2. 結果が不十分ならTesseract.jsへフォールバック
3. 元画像、EXIF Orientation補正、中央切り出し、拡大グレー、コントラスト、二値化、シャープ化を生成
4. Tesseract.jsで複数前処理を順にOCR
5. 信頼度、文字量、候補一致数、前処理種別ボーナスでスコアリング
6. 最良結果を採用
7. 銘柄候補は完全一致、部分一致、誤認識補正、類似度、蔵元、容量、酒種で評価
8. 自動確定せず、ユーザー確認を必須にする

## 実装した代替案

- OCRが低精度な場合: 複数前処理のTesseract.js再試行
- 銘柄が読めない場合: 蔵元名、容量、英語名、表記ゆれ、過去候補との整合
- HEICが扱えない場合: heic2anyでJPEG変換を試行し、失敗時はJPEG変換方法を案内
- 価格が取れない場合: 検索語パターン変更、過去履歴、手入力、未取得保存
- 誤候補防止: OCR結果が空の場合は固定有名銘柄を表示しない

## 市場価格照合の反復検証

楽天アプリIDは秘密情報にあたるため、実APIキーを使う検証は実施していません。代替として `tests/results/price-results.json` に検索語パターンと候補スコアリングのフィクスチャ検証を保存しました。

比較した検索語:

- 銘柄名のみ
- 銘柄名＋蔵元
- 銘柄名＋容量
- 銘柄名＋蔵元＋容量
- OCR原文
- 正規化後の銘柄名
- 英語名
- 別名・略称

検証した除外・警告:

- 容量違い
- 本数違い
- セット販売
- ギフトセット
- ふるさと納税
- 送料別または送料不明
- 同名別商品

## 残る制約

- 実写真ではなく合成画像中心の検証であり、瓶の曲面、液面反射、ラベル紙質、背景の複雑さは完全再現できない
- iPhone Safari実機、Android Chrome実機での実操作確認はこの環境では未実施
- WebPとHEICの実ファイルOCRはブラウザ実装側での対応であり、今回のCLI検証ではHEIC由来相当JPEGで代替
- 筆文字風の銘柄はOCR単体では完全一致しにくく、蔵元・英語名・類似度補正との組み合わせが必要
